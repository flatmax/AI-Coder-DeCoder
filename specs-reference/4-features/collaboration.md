# Reference: Collaboration

**Supplements:** `specs4/4-features/collaboration.md`

This is the canonical owner for admission message types, timeout values, and the share-info payload. Pre-JRPC handshake messages use raw WebSocket text frames, not JSON-RPC — the server hasn't created a JRPC remote for the pending client yet.

## Byte-level formats

### Raw WebSocket admission messages (pre-JRPC)

Three message types sent on the raw WebSocket before JRPC setup, for clients waiting on admission. Each is a JSON object with a `type` field and message-specific payload.

**`admission_pending`** — sent immediately when a non-first connection arrives:

```json
{"type": "admission_pending", "client_id": "uuid-string-generated-server-side"}
```

The `client_id` is a UUID4 hex string (no hyphens) generated by the server. The client holds this ID for the duration of the wait; it becomes the client's JRPC remote UUID if admission succeeds.

**`admission_granted`** — sent when a pending client is admitted:

```json
{"type": "admission_granted", "client_id": "same-uuid-as-pending"}
```

After the client receives this, the server proceeds with normal JRPC handshake. The `client_id` field is for correlation — the client may have multiple tabs open, each with its own pending state.

**`admission_denied`** — sent just before the WebSocket is closed with code 1008:

```json
{"type": "admission_denied", "client_id": "same-uuid-as-pending", "reason": "Denied by host"}
```

The `reason` field is a human-readable string. Current values:

- `"Denied by host"` — an admitted user clicked Deny
- `"Request timed out"` — 120-second admission window elapsed without action
- `"Replaced by newer request from same IP"` — a new connection arrived from the same IP with a previous request still pending

### WebSocket close codes

After `admission_denied`, the server closes with code `1008` (Policy Violation) and empty reason. Code 1008 is the WebSocket standard for "policy did not permit this connection" — appropriate for the admission denial case.

The admission-timeout and same-IP-replacement cases also close with 1008 after their respective `admission_denied` messages.

### Frontend interception

The webapp intercepts raw WebSocket messages before jrpc-oo processes them. An `addEventListener('message', listener, true)` in capture phase inspects each incoming frame's text content and checks for the three admission message types by parsing as JSON and checking `data.type`:

```javascript
ws.addEventListener('message', (event) => {
  try {
    const data = JSON.parse(event.data);
    if (data.type === 'admission_pending' ||
        data.type === 'admission_granted' ||
        data.type === 'admission_denied') {
      event.stopImmediatePropagation();
      // ... handle admission flow ...
    }
  } catch {
    // not JSON or not admission — let jrpc-oo handle
  }
}, true);  // capture phase
```

The three type strings are matched exactly. Any other message format (including JRPC frames) falls through to jrpc-oo's own handler.

### RPC broadcast events

After admission, state changes are broadcast via standard JRPC. Four event types, each a method call from server to all admitted remotes.

**`AcApp.admissionRequest(data)`** — new connection pending, admitted users should show admission toast:

```json
{
  "client_id": "uuid-string",
  "ip": "192.168.1.42",
  "requested_at": "2025-01-15T10:30:00.123456+00:00"
}
```

**`AcApp.admissionResult(data)`** — pending request resolved, dismiss toast:

```json
{
  "client_id": "uuid-string",
  "ip": "192.168.1.42",
  "admitted": true,
  "replaced": false
}
```

Field semantics:
- `admitted: true` — host clicked Admit, the pending client is now a full participant
- `admitted: false, replaced: false` — host clicked Deny, the pending client is being disconnected
- `admitted: false, replaced: true` — new request from same IP superseded this one (auto-deny, no user action)

**`AcApp.clientJoined(data)`** — admitted client completed JRPC handshake:

```json
{
  "client_id": "uuid-string",
  "ip": "192.168.1.42",
  "role": "participant",
  "is_localhost": false
}
```

**`AcApp.clientLeft(data)`** — client disconnected:

```json
{
  "client_id": "uuid-string",
  "ip": "192.168.1.42",
  "role": "participant"
}
```

**`AcApp.roleChanged(data)`** — sent to one specific client when their role changes (e.g., promoted to host after previous host disconnected):

```json
{
  "role": "host",
  "reason": "previous host disconnected"
}
```

Currently the only `reason` value is `"previous host disconnected"`. Future expansion might include `"manual promotion"` or similar.

## Schemas

### `Collab.get_share_info()` return shape

Returns the payload the browser needs to build a share URL for collaborators to visit:

```json
{
  "ips": ["192.168.1.50", "10.0.0.3"],
  "port": 18999
}
```

- `ips` — list of routable LAN IPs. Loopback (`127.*`), IPv6 loopback (`::1`), `localhost`, and link-local IPv6 (`fe80::*`) addresses are filtered out. Remote collaborators cannot reach those
- `port` — the webapp HTTP port (not the WebSocket port). The share URL has the form `http://{ip}:{port}/?port={ws_port}` — the webapp reads the WebSocket port from the query parameter

When no routable IPs are available (pure loopback environment), `ips` is an empty list. The frontend should show an instructional message rather than a copyable URL.

### `Collab.get_collab_role()` return shape

Called by a client after JRPC setup to learn its own identity:

```json
{
  "role": "host",
  "is_localhost": true,
  "client_id": "uuid-string"
}
```

### `Collab.get_connected_clients()` return shape

```json
[
  {
    "client_id": "uuid-1",
    "ip": "127.0.0.1",
    "role": "host",
    "is_localhost": true
  },
  {
    "client_id": "uuid-2",
    "ip": "192.168.1.42",
    "role": "participant",
    "is_localhost": false
  }
]
```

Ordered by `admitted_at` timestamp ascending (host first, newest participant last).

### Restricted error shape

See `specs-reference/1-foundation/rpc-inventory.md` § Restricted error shape for the canonical `{"error": "restricted", "reason": "..."}` format used by every mutating method when called by non-localhost participants. Collaboration does not introduce a separate error format; it reuses the common one.

## Numeric constants

### Admission timeout

```
120 seconds
```

Pending requests not acted on within this window are auto-denied. The server uses `asyncio.wait()` with `timeout=120.0` and `return_when=FIRST_COMPLETED` to race the admit/deny future against the WebSocket's `wait_closed` future.

Tuning note: 120 seconds balances "host noticed the toast, read the IP, decided" against "forgotten pending request blocks a zombie WebSocket forever." Users wanting stricter policies can enforce via external firewall rules.

### Same-IP replacement

```
1 pending request per IP
```

When a new connection arrives from an IP already in the pending queue, the older request is auto-denied with `replaced: true` before the new request is added. Handles browser refresh cases where the page reconnects before the user has clicked Admit on the first attempt.

IP comparison is exact string match on the peer IP (from `websocket.remote_address[0]`). IPv4-mapped IPv6 addresses (`::ffff:192.168.1.1`) compare differently from their IPv4 form (`192.168.1.1`) — this is acceptable because browser connections consistently produce one form or the other within a session.

### WebSocket close code for denial

```
1008
```

Policy Violation per RFC 6455. Applied to all three denial cases (explicit deny, timeout, same-IP replacement) after the corresponding `admission_denied` message is sent.

### Broadcast retry

Broadcasts use the standard RPC call mechanism; no additional retry logic at the collaboration layer. Delivery failures (client disconnected mid-broadcast) are logged but not retried — the `clientLeft` event that follows is sufficient to reconcile state on other clients.

## Schemas

### Client registry entry (internal)

Each admitted client is stored server-side as:

```pseudo
ConnectedClient:
    client_id: string       # UUID hex
    ip: string              # Peer IP from websocket.remote_address[0]
    role: "host" | "participant"
    is_localhost: bool
    admitted_at: float      # time.time() at admission
    websocket: WebSocket    # reference for direct send if needed
    remote: JRPCRemote      # the jrpc-oo remote for this client
```

Not exposed via RPC — `get_connected_clients()` returns a projection without `websocket` or `remote`.

### Pending request entry (internal)

Separate from the admitted-client registry:

```pseudo
PendingRequest:
    client_id: string
    ip: string
    websocket: WebSocket    # pre-JRPC raw connection
    future: asyncio.Future  # resolved by admit/deny or timeout
    requested_at: float
```

No `remote` field because no JRPC remote has been created yet.

### Localhost detection

A peer IP counts as localhost if it matches ANY of:

1. Loopback IPv4 (`127.0.0.0/8`)
2. Loopback IPv6 (`::1`)
3. An IP address assigned to a local network interface (from `socket.gethostbyname_ex(socket.gethostname())` and similar enumeration)

Interface enumeration catches the case where the host opens their browser to their LAN IP (e.g., `192.168.1.50`) rather than `localhost`. A connection from that LAN IP to the same machine is treated as localhost.

The detection result is computed once at connection time and cached on the `ConnectedClient` record. It does not update if network interfaces change mid-session.

## Dependency quirks

### `websockets` library peer address format

`websocket.remote_address` is a tuple of `(host, port)` in IPv4 or `(host, port, flowinfo, scope_id)` in IPv6. The collaboration layer takes only the first element (`[0]`). For IPv6 clients, this gives the plain address without zone identifiers — suitable for display and comparison but not for reconstructing a connect-back URL (which would need the full tuple).

### Same-origin behavior

When a collaborator opens the share URL, they visit the webapp from the host's LAN IP. The webapp's JavaScript then constructs the WebSocket URL from `window.location.hostname` — this correctly targets the same LAN IP, not `localhost`.

Clients that visit the webapp via `localhost` (the host's own browser, or a tunnel with `localhost` rewriting) connect their WebSocket to `localhost`, which the server sees as a loopback connection. Both cases work correctly.

### First-connection auto-admit is unconditional

The first connection after server startup is always auto-admitted, regardless of peer IP. Even a non-localhost first connection becomes host. This is a feature, not a bug — it allows remote setups where the server has no local browser client (CI, headless development box) to bootstrap without needing a separate localhost admit step.

If this behavior is a concern for a given deployment, the recommended path is starting the server with `--collab` only after a trusted localhost browser has connected. The server binds to `0.0.0.0` only when `--collab` is passed; without the flag, non-localhost connections are impossible at the TCP level.
